# RFC 016 — Server logging completeness: request tracing, file output, dev defaults

**Status.** Proposed
**Priority.** High. Sequenced after RFCs 010 / 011 / 012; ahead of
RFC 003 if the implementer prefers, since this work makes
debugging the others easier.
**Tracks.** v0.29.3 maintainer query — "is server logging
sufficient?" surfaced two concrete gaps: a dev-mode operator can't
confirm requests are arriving, and there is no file-based logger.
**Touches.** `crates/sui-id/src/startup.rs` (init_tracing extended
with a file appender), `crates/sui-id/src/router.rs` (TraceLayer
mounted), `crates/sui-id/src/config.rs` (LogConfig gains opt-in
fields), `crates/sui-id/src/dev_mode.rs` (different defaults),
`crates/sui-id-store` (audit hashing of redacted log invariants —
see Tests), `Cargo.toml` (`tracing-appender`, `tower-http` `trace`
feature).

## Summary

sui-id today has `tracing` + `tracing-subscriber` wired in and
emits warnings/errors at handler-level for known failure
branches. What it lacks is the routine signal an operator needs
to confirm the server is actually doing work: HTTP requests
coming in, basic responses going out, and a file the operator
can `tail -f` after the fact.

The router file even acknowledges this — there's a comment
referring to "TraceLayer's span" that no `TraceLayer` is mounted
to produce. Today the only HTTP-correlated identifier is the
`X-Request-Id` header from the request-id middleware, which is
generated and propagated correctly but never logged because
nothing logs at the request boundary.

This RFC closes the gap with a tight, security-conscious design:
add `TraceLayer`, add a file appender behind opt-in config,
default access logging on in dev mode, default it off in
production, and pin the redaction invariant — passwords, codes,
tokens, cookie values must never reach any sink — with explicit
tests.

## Why this is high-priority

The maintainer query landed two findings worth treating as
release blockers for the next implementation pass:

- **Operability gap.** Dev-mode operators cannot verify that
  requests are reaching sui-id at all. A new contributor (or
  the maintainer themselves, after a few weeks away) hits the
  process with `curl` and gets no signal. This is the first
  rung on the debugging ladder; without it, every other
  RFC's implementer is debugging blind.
- **Audit / forensic gap.** Production deployments today have
  to choose between "log nothing useful" and "configure
  external syslog redirection." A file appender that's off
  by default but available with one config field is the
  shape that matches sui-id's "minimum operating ceremony"
  posture — present when wanted, absent when not.

Sequenced after RFC 010 (forgot-password revoke), 011
(WebAuthn transport), and 012 (setup wizard reconciliation),
because those three are smaller and more critical. Sequenced
ahead of RFC 003 if the implementer prefers — having request
logs working makes implementing the HIBP scope expansion
materially easier to verify.

## Requirements

After this RFC ships:

1. In dev mode, every incoming HTTP request produces at least
   one log line at `INFO` level with method, path, status, and
   request-id. Operator running `sui-id --dev` can `curl
   localhost:8801/healthz` and see the request hit.
2. In production mode (default config), HTTP request logging
   is *off*. The existing warn/error logging from handler
   error branches is unchanged.
3. Production deployments can opt into request logging with
   one config field (`log.access_log = true`). When enabled,
   the format is the same as dev's.
4. A new optional config field `log.file = "..."` directs all
   tracing output to a rolling file instead of stdout. Default
   absent — stdout-only. When set, daily rotation, retention
   bounded by configuration.
5. **Redaction invariant**: passwords (plaintext or hashed),
   TOTP codes, recovery codes, authorization codes, refresh
   tokens, access tokens, master keys, session cookie values,
   reset tokens, WebAuthn challenge bytes — none of these ever
   reach any logging sink. Whatsoever. Tested explicitly and
   regression-tested.
6. `X-Request-Id` (already generated by existing middleware)
   appears in every request log line, threading from middleware
   through handler-emitted log events on the same request.
7. Existing audit log behaviour is unchanged. The audit log is
   *not* a logging concern; it's an integrity-protected event
   stream.
8. `--dev` startup banner is unchanged in shape but gains one
   line: "Access log: on (dev default). Use `log.access_log =
   false` in config to suppress."

## Design

### Library additions

Two crates:

- `tower-http` already in workspace dependencies (used for
  CORS); add the `trace` feature to enable `TraceLayer`.
- `tracing-appender = "0.2"` (new). Provides
  `RollingFileAppender` and the `non_blocking` writer wrapper
  needed to avoid synchronous file I/O on the request thread.

Both are mainstream Tokio-ecosystem crates. No new transitive
audit surface beyond what's already there.

### `TraceLayer` mounting

In `crates/sui-id/src/router.rs`, after the existing
`request_id::middleware` layer, add `TraceLayer::new_for_http()`
configured to use the request-id from headers:

```rust
.layer(TraceLayer::new_for_http()
    .make_span_with(|req: &Request<_>| {
        let request_id = req
            .headers()
            .get("x-request-id")
            .and_then(|v| v.to_str().ok())
            .unwrap_or("-");
        tracing::info_span!(
            "http",
            method = %req.method(),
            path = %req.uri().path(),
            request_id = %request_id,
        )
    })
    .on_request(())                        // suppress separate request log
    .on_response(DefaultOnResponse::new()
        .level(tracing::Level::INFO)
        .latency_unit(LatencyUnit::Millis))
    .on_failure(()))                       // suppress 5xx auto-log; handler logs it
```

Notes:
- `make_span_with` extracts the request-id that's already been
  attached by the existing middleware. `request_id::middleware`
  must run *before* `TraceLayer` so the header is present —
  current ordering already has it as the outermost layer, which
  is correct.
- `on_request(())` suppresses the per-request log (we get the
  start info from the span itself).
- `on_response(...)` emits one line per response with the
  status and latency.
- `on_failure(())` suppresses tower-http's automatic 5xx log;
  handler-level `tracing::error!` calls already cover those
  with more useful context.

The query string is **not** logged. URLs containing
`code=...`, `state=...`, or query parameters with PKCE values
must not have those values printed. Path-only is sufficient
for routine ops; full URLs are available in the warn/error
logs from handlers when they matter.

### File appender wiring

Extend `init_tracing()` in `startup.rs`:

```rust
pub fn init_tracing(cfg: &LogConfig) {
    INIT_TRACING.call_once(|| {
        let env_filter = EnvFilter::try_new(&cfg.filter)
            .unwrap_or_else(|_| EnvFilter::new("info"));

        let (writer, _guard) = match &cfg.file {
            Some(path) => {
                let file_appender = tracing_appender::rolling::daily(
                    path.parent().unwrap_or(Path::new(".")),
                    path.file_name()
                        .and_then(|s| s.to_str())
                        .unwrap_or("sui-id.log"),
                );
                let (nb, guard) = tracing_appender::non_blocking(file_appender);
                (BoxMakeWriter::new(nb), Some(guard))
            }
            None => (BoxMakeWriter::new(std::io::stdout), None),
        };

        // The non-blocking guard must outlive the program. Stash it.
        if let Some(g) = _guard {
            FILE_APPENDER_GUARD.set(g).ok();
        }

        let builder = tracing_subscriber::fmt()
            .with_env_filter(env_filter)
            .with_target(true)
            .with_level(true)
            .with_writer(writer);

        if cfg.format == "json" {
            builder.json().init();
        } else {
            builder.init();
        }
    });
}
```

The `_guard` pattern is `tracing_appender`'s
required-but-easy-to-miss invariant: dropping the guard
flushes the buffered writer, so it has to live as long as the
process.

Rotation: daily by default. The
`tracing_appender::rolling::daily` constructor is the
opinionated "rotate at midnight UTC, keep yesterday's file
named with `.YYYY-MM-DD` suffix" shape. We deliberately do
**not** add hourly or size-based rotation; daily is enough
for sui-id's expected scale.

Retention: not enforced by sui-id. The operator's logrotate
or equivalent can clean up old files. Documenting this in
`docs/operators.md` is part of RFC 015's pass.

### Config additions

`LogConfig` gains two optional fields:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct LogConfig {
    /// `fmt` (human-readable) or `json` (one JSON object per line).
    #[serde(default = "default_log_format")]
    pub format: String,
    /// `tracing-subscriber` env-filter expression.
    #[serde(default = "default_log_filter")]
    pub filter: String,
    /// If set, log lines go to a daily-rotated file at this path
    /// instead of stdout. Default absent (stdout-only).
    #[serde(default)]
    pub file: Option<PathBuf>,
    /// Whether to log every HTTP request/response at INFO. Default
    /// false in production, overridden to true in `--dev`.
    #[serde(default)]
    pub access_log: bool,
}
```

Default values keep current behaviour for existing
deployments: stdout-only, no access log, filter `info`.

### Dev mode override

In `dev_mode.rs`, after parsing the dev config, set:

```rust
cfg.log.access_log = true;
```

…before `init_tracing`. The startup banner gains:

```
DEV MODE — access log: ON (dev default).
            Set `log.access_log = false` in config to suppress.
```

If the operator's dev-config TOML explicitly sets
`log.access_log = false`, that's honoured — the override is
"dev defaults *to* on," not "dev forces on."

### Redaction invariant

This is the security-critical part. The RFC pins:

**Never logged at any level**:
- Plaintext passwords (any handler that takes one)
- TOTP codes (six-digit verification values)
- Recovery codes
- Authorization codes (the `code` parameter at `/oauth2/authorize`'s redirect)
- Access tokens, refresh tokens, ID tokens (full JWT contents)
- Master key bytes
- Session cookie values (the random session ID after `=`)
- CSRF cookie values
- Password reset tokens (the URL-safe-base64 token)
- WebAuthn challenge bytes, attestation data
- HIBP-checked password prefixes (the 5-char SHA-1 prefix
  is sent to HIBP, never logged locally)
- SMTP authentication credentials

**Allowed at INFO** (no secret content):
- HTTP method, path (no query string)
- Response status, latency
- Request-id
- Username (the *display* identifier — not the password)
- Client ID (UUID, not secret)
- User ID
- Audit event names (already public-by-design)

**Allowed at WARN/ERROR**:
- Same as INFO plus error-class identifiers and the kind of
  failure (e.g. "wrong_password", "csrf_mismatch")
- Never the value that failed verification

**Allowed at DEBUG/TRACE** (developer-only, gated behind
explicit env-filter override):
- More verbose internal state, but still no secrets — the
  invariant doesn't relax with log level.

### Implementation: how to enforce redaction

Three mechanisms, layered:

1. **No `Display` / `Debug` for secret types.** sui-id already
   has `#[derive(Debug)]` on most structs but the secret-bearing
   types (`Password`, `MasterKey`, `RecoveryCode`, etc) either
   already have hand-rolled `Debug` impls that print `"REDACTED"`
   or should after this RFC. The audit pass: every type holding
   a secret has a custom `Debug` returning `"REDACTED"`. No
   `#[derive(Debug)]` on those.
2. **`secrecy::Secret<T>` for in-flight secrets.** Already in
   workspace dependencies (used for the master key). Extend to
   wrap forms' password fields, TOTP codes, etc. The
   `Secret<String>` wrapper makes it a compile error to print
   the contained value without explicit `expose_secret()`.
3. **Test fixtures that grep the log buffer.** A handful of
   `tests/e2e/redaction.rs` tests drive specific flows
   (login, password change, token exchange) with a captured
   log writer and grep for known secret values to ensure they
   don't appear. See Tests below.

### Audit log boundary

The audit log is fundamentally a *different* thing from
operational logging. To restate the invariant:

| Concern | Audit log | Tracing log |
|---|---|---|
| Storage | SQLite with hash chain | stdout or file |
| Retention | Indefinite, prune deliberate | Operator's logrotate |
| Integrity | SHA-256 chain | None |
| Queryable | SQL, admin UI | grep / log aggregator |
| Contains secrets | Never | Never |
| Required for compliance | Yes | No |

This RFC does not change the audit log. Tracing-log lines
that reference an audit event (e.g.
`tracing::info!(event = "auth.login.success", user_id = %u)`)
are *informational* — the audit row is the authoritative
record.

## Multiple implementation steps

Each step is independently shippable.

1. **TraceLayer mounted; access_log config added.** Default
   off in production, on in dev. No file appender yet. This
   is the smallest change that closes the operator-visibility
   gap from the maintainer query.
2. **File appender.** `log.file` config field. Daily rotation.
   Documentation pass on logrotate guidance.
3. **Redaction invariant audit + tests.** Walk every secret
   type, ensure custom `Debug`. Add `tests/e2e/redaction.rs`.
   This is the largest single piece of work in the RFC and
   the most security-relevant.
4. **JSON format polish.** Verify the JSON output is sane
   when access_log is on (request span keys, latency unit,
   etc). The `json` format flag exists; this just confirms
   it produces useful structure.

In `crates/sui-id-core/src/<module>/tests.rs` for each

```rust
#[test]
fn debug_does_not_leak_password() {
    let p = Password::new("alice-the-tester-password");
    assert_eq!(format!("{p:?}"), "Password(REDACTED)");
    assert_eq!(format!("{p:#?}"), "Password(REDACTED)");
}
```

### E2E redaction

A new file `tests/e2e/redaction.rs` mounts a custom
`tracing_subscriber` writer that captures all output to a
`Vec<u8>`, drives representative flows, and asserts known
secret values don't appear:

```rust
#[tokio::test]
async fn login_does_not_log_password() {
    let (state, captured) = test_app_with_captured_logs();
    complete_setup_and_login(&state).await;

    let logs = captured.lock().unwrap();
    let logs_str = String::from_utf8_lossy(&logs);
    assert!(
        !logs_str.contains(PASSWORD),
        "logs leaked the password: {logs_str}"
    );
}

#[tokio::test]
async fn token_exchange_does_not_log_authorization_code() { ... }

#[tokio::test]
async fn refresh_rotation_does_not_log_refresh_token() { ... }

#[tokio::test]
async fn forgot_password_does_not_log_reset_token() { ... }

#[tokio::test]
```

`test_app_with_captured_logs()` is a new helper in

## Security considerations

- **The redaction invariant is the entire point.** A regression
  here is worse than no logging at all — operators trust log
  files to be safe to share with support, copy to their issue
  tracker, or ship to a log aggregator. A leaked password in
  any of those venues is a breach. The test suite must keep
  the invariant honest.
- **Default off in production.** Even with redaction, an
  attacker who reaches the log file learns who logged in when
  and from where (via the existing X-Forwarded-For handling).
  That's metadata, but it's metadata. Operators opt in
  deliberately.
- **File appender failure modes.** A full disk fills the
  non-blocking writer's buffer; once the buffer is full,
  log lines are *dropped* rather than blocking the request
  thread. This is the right trade-off for a security-
  sensitive service: a backed-up logger must not become a
  DoS vector.
- **Log injection.** Path values and usernames are passed to
  the formatter. The default `tracing_subscriber` `fmt` layer
  escapes newlines; the `json` layer escapes them via JSON
  encoding. We don't construct logged messages by string
  concatenation — every variable is a structured field.
- **External log aggregator targeting.** Out of scope for this
  RFC. An operator who wants Loki / Splunk / Datadog can pipe
  stdout to the agent of their choice, or `tail -f` the
  appender file. We don't ship integration code for any
  specific aggregator.
- **dev-mode credentials.** The dev banner already prints the
  hardcoded admin/alice/bob passwords to stderr — that's
  spec-mandated and outside this RFC. Access log lines
  during dev sessions don't add to that exposure (they
  contain no password content).
- **Request body logging.** Explicitly **not** done. Even with
  redaction, logging request bodies risks leaking content the
  user didn't consent to operationally exposing. Out of scope.

## Open questions

- **Should the access log respect a per-route allowlist** (e.g.
  log `/oauth2/token` requests but not `/static/*`)?
  Recommend no for v1 — adds complexity, the static-asset
  noise is small, and an operator who cares can grep. Revisit
  if real deployments produce noisy logs.
- **Should there be a `log.access_log_excludes` config field**
  for explicit per-path suppression (e.g. `/healthz` for
  liveness probes)?
  Recommend yes. Healthz spam is the most common request to
  filter out. One field, list of path prefixes, default
  `["/healthz"]`. Folded into step 1.
- **Should the file appender support compression**
  (`tracing-appender` doesn't, but a wrapper crate does)?
  Recommend no. Operator's logrotate already does this. Keep
  the in-process logger simple.
- **Should we log to syslog directly?**
  Recommend no. Stdout + an external syslog forwarder is the
  standard Unix shape. Adding a sui-id-internal syslog client
  pulls in a dependency and an OS coupling for a feature that
  external tooling already covers.
